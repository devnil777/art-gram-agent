# Техническое задание

## 1. Цель

Разработать модуль сбора данных из Telegram-каналов через Telethon, запускаемый внешним cron, сохраняющий сырые сообщения в файловое хранилище, и обеспечивающий надежный инкрементальный сбор без повторного обращения к уже собранным данным. Сборщик должен быть однопоточным, отказоустойчивым, идемпотентным и пригодным для последующей агрегации через LLM.

## 2. Общий принцип архитектуры

Система должна быть разделена на два независимых слоя:

- слой сбора и хранения сырых данных;
- слой обработки и агрегации.

Сборщик не должен содержать бизнес-логику анализа сообщений. Его задача — только извлечь, нормализовать и сохранить данные в прозрачную файловую структуру. Все данные должны сохраняться в append-only формате, чтобы исключить перезапись истории и снизить риск потери данных.

## 3. Область ответственности блока

Блок должен выполнять следующие функции:

- читать конфигурацию;
- автоматически получать список всех каналов, доступных текущему пользователю в Telegram;
- читать текущее состояние по каждому каналу;
- забирать новые сообщения из всех доступных Telegram-каналов через Telethon;
- сохранять сырые сообщения в JSONL-файлы;
- обновлять чекпоинт по каналу только после успешной записи данных;
- вести журнал запусков и ошибок;
- готовить артефакты с описанием структуры, форматов и порядка работы.

## 4. Ограничения

- Сбор однопоточный.
- Источник данных на первом этапе только Telegram channels.
- Запуск процесса происходит посредством внешнего cron, например каждые 4 часа.
- Хранение данных рассчитано на небольшой объем, до 30 каналов и 1–2 недели истории.
- Не использовать базу данных.
- Все сообщения в коде, логах и пользовательских скриптах должны быть на английском языке.
- В коде запрещены Unicode-символы в текстовых строках сообщений, логах и именах технических сущностей.
- Кодировка всех файлов проекта и создаваемых артефактов должна быть UTF-8.

## 5. Источник данных

Telegram-каналы должны читаться через Telethon. Для инкрементального доступа допускается использовать историю сообщений с курсором по `message_id` и/или `offset_id`, так как Telethon поддерживает выборку истории с пагинацией.
Сборщик должен ориентироваться на последний гарантированно сохраненный `message_id` по каждому каналу, а не только на временной интервал.

## 6. Файловая структура хранения

Реализовать следующую структуру каталогов:

- `config/` — конфигурационные файлы.
- `state/` — состояние по каналам.
- `raw/` — сырые сообщения в JSONL.
- `runs/` — журналы запусков.
- `reports/` — итоговые отчеты и артефакты описания.
- `logs/` — текстовые логи приложения.
- `schemas/` — описание форматов файлов и структуры данных.

Пример:

- `config/app.yaml`
- `config/channels.yaml`
- `state/channel_<id>.json`
- `raw/channel_<id>/YYYY-MM-DD.jsonl`
- `runs/run_<timestamp>.json`
- `logs/collector.log`
- `reports/structure.md`
- `reports/data_dictionary.md`
- `schemas/raw_message.schema.json`

## 7. Формат конфигурации

Конфигурация должна задаваться через файл, без хардкода параметров.

Минимальный набор параметров:
- `telegram.api_id`
- `telegram.api_hash`
- `telegram.session_name`
- `collector.cron_interval`
- `collector.batch_size`
- `collector.overlap_messages`
- `collector.initial_search_depth` (количество сообщений для новых каналов)
- `collector.retry_count`
- `collector.retry_backoff_seconds`
- `storage.base_path`
- `storage.raw_format = jsonl`
- `logging.level`
- `logging.file_path`

Список каналов получается автоматически через API Telegram. Опционально может использоваться файл исключений или фильтрации, если это предусмотрено реализацией.

ТЗ требует, чтобы настройки были документированы в отдельном артефакте с описанием каждого параметра, допустимых значений и поведения по умолчанию.

## 8. Формат хранения сырых сообщений

Каждая строка JSONL должна представлять одно сообщение Telegram. Для каждого сообщения сохранять минимум:

- `channel_id`
- `channel_username`
- `message_id`
- `message_date`
- `text`
- `raw`
- `is_forwarded`
- `reply_to_message_id`
- `edit_date`
- `ingested_at`
- `run_id`

Требования:
- формат JSONL;
- одна строка = одно сообщение;
- файл должен дополняться, а не переписываться целиком;
- записи должны быть пригодны для повторного чтения и агрегации;
- при необходимости сохранять полный сырой объект Telethon в поле `raw`.

## 9. Чекпоинт и состояние

Для каждого канала должен существовать отдельный state-файл. В нем хранить:

- `channel_id`
- `last_success_message_id`
- `last_success_message_date`
- `last_success_run_id`
- `last_attempt_at`
- `last_success_at`
- `error_count`
- `status`

Правило обновления состояния:
- сначала успешно сохранить raw-данные;
- только затем обновить state;
- при ошибке state не должен продвигаться вперед.

Это обязательное условие для минимизации потери данных и обеспечения идемпотентности.

## 10. Алгоритм сбора

Сборщик должен работать по следующему циклу:

1. Загрузить конфиг.
2. Получить из Telegram список всех доступных каналов пользователя (диалоги типа channel).
3. Для каждого канала прочитать state.
4. Определить стартовую точку сбора по `last_success_message_id`. Если канал «новый» (состояние отсутствует), глубина поиска задается в настройках (`initial_search_depth`).
5. Запросить Telegram историю пачками.
6. Нормализовать сообщения.
7. Записать их в временный файл или временный буфер.
8. После успешной записи атомарно зафиксировать результат.
9. Обновить state только после подтвержденной записи.
10. Записать результат в run log.
11. Перейти к следующему каналу.

При любом сбое сборщик должен:
- сохранить факт ошибки;
- не потерять уже записанные данные;
- не повреждать существующие raw-файлы;
- корректно завершить текущий канал и продолжить по остальным, если это безопасно.

## 11. Надежность записи

Для файловых операций использовать атомарный подход:
- писать во временный файл;
- после успешной записи выполнять atomic rename / replace;
- по возможности делать fsync перед финализацией.

Если запись не завершена, итоговый файл не должен оказаться частично поврежденным. Такой способ соответствует best practice для JSONL и других файловых форматов. 

## 12. Логирование

Нужно реализовать отдельное логирование приложения.

Требования:
- все лог-сообщения на английском языке;
- без Unicode-символов;
- формат логов — text/plain;
- уровень логирования задается в конфиге;
- лог должен содержать timestamp, level, module, run_id, channel_id, message;
- ошибки должны логироваться с контекстом канала и последней успешной точки.

Логи должны помогать быстро ответить на вопросы:
- какой канал обрабатывался;
- сколько сообщений собрано;
- на каком сообщении произошел сбой;
- что было обновлено;
- был ли повторный запуск.

## 13. Артефакты, которые должны быть созданы

Это обязательная часть работы. После реализации ИИ-агент должен создать набор артефактов, которые подробно описывают реализованную структуру и поведение.

### Обязательные артефакты
- `reports/structure.md`
- `reports/configuration.md`
- `reports/logging.md`
- `reports/storage_format.md`
- `reports/collection_algorithm.md`
- `reports/reliability.md`
- `reports/data_dictionary.md`
- `schemas/raw_message.schema.json`
- `schemas/channel_state.schema.json`
- `schemas/run_log.schema.json`

### Что должно быть в артефактах
- полное описание директорий и файлов;
- описание конфигурации и всех параметров;
- описание структуры raw JSONL;
- описание state-файлов;
- описание алгоритма сбора;
- описание retry и error handling;
- описание logging policy;
- описание инвариантов целостности данных;
- примеры файлов и примеры записей;
- пояснение, как работает идемпотентность;
- пояснение, как обеспечивается recovery после сбоя.

## 14. Требования к коду

- Язык реализации: Python.
- Интеграция с Telegram: Telethon.
- Код должен быть структурирован по модулям.
- Все строки пользовательских сообщений и технических логов в коде — на английском языке.
- В коде не использовать Unicode-символы в строках сообщений, логах, именах ключей и служебных идентификаторах.
- Все файлы, создаваемые кодом, должны быть в UTF-8.
- Код должен быть читаемым, с минимально достаточными комментариями.
- Конфигурация должна загружаться из файла.
- Пути должны быть параметризованы через конфиг.
- Не допускать hardcode значений, влияющих на runtime поведение.

## 15. Критерии приемки

Реализация считается завершенной, если:

- сборщик читает каналы по расписанию;
- новые сообщения сохраняются в JSONL;
- state обновляется только после успешной записи;
- повторный запуск не создает дубликаты;
- при ошибке Telegram данные не теряются;
- логи создаются и читаются;
- все сообщения в коде и логах на английском языке;
- все текстовые файлы используют UTF-8;
- созданы все обязательные артефакты;
- артефакты подробно описывают структуру, конфиг, логи, формат хранения и надежность.

## 16. Необходимые инварианты

Система должна гарантировать:
- raw-данные неизменяемы после записи;
- state никогда не опережает реально записанные данные;
- `message_id` уникален внутри канала;
- повторные попытки безопасны;
- пропуск запуска не приводит к потере данных;
- сбой в середине цикла не ломает уже сохраненную историю.

## 17. Рекомендуемый формат сдачи

ИИ-агент должен сдать:
- исходный код;
- конфиг-файлы;
- схему каталогов;
- артефакты описания;
- примеры generated files;
